# gateway/nodes: `maybeWakeNodeWithApns` 가 registration 체크 전 `nodeWakeById.set` 수행 → 미등록 nodeId 누수 (no-registration 분기 cleanup 추가)

## 문제 요약

`src/gateway/server-methods/nodes.ts:308-416` 의 `maybeWakeNodeWithApns` 는
진입 즉시 `nodeWakeById.set(nodeId, state)` 를 수행한다 (L312-313). 이 set 은
`state.inFlight` 를 이용한 concurrent wake 호출 coalescing 을 위한 것이며,
`loadApnsRegistration(nodeId)` 은 그 뒤 L333 에서야 호출된다. 미등록 nodeId 에
대해 `loadApnsRegistration` 이 `null` 을 반환하면 L334-336 에서
`path: "no-registration"` 으로 조기 반환하지만, 이 경로에서 `nodeWakeById`
엔트리를 삭제하는 cleanup 은 부재다.

엔트리의 유일 삭제 경로는 `clearNodeWakeState` (L525-528), 유일 호출자는
`src/gateway/server/ws-connection.ts:327` — WS close + `client?.connect?.role === "node"`
+ `nodeRegistry.unregister(connId)` 가 유효 nodeId 를 반환할 때만 fire. 즉
**WS 연결 cycle 을 경험한 적 없는 nodeId** (오탈자 / 재페어링 후 stale /
APNs 등록이 clearStaleApnsRegistrationIfNeeded 로 이미 제거된 경우) 에 대해서는
cleanup 이 영원히 fire 하지 않는다.

operator 가 `node.pending.enqueue` (nodes-pending.ts:89) 또는 `node.invoke`
(nodes.ts:920, 943, 1050) 를 arbitrary nodeId 로 호출할 수 있고 — validator
`validateNodePendingEnqueueParams` / `validateNodeInvokeParams` 는 type-check
만 수행 — paired device allowlist 교차검증이 없으므로 leak 경로는 realizable.

엔트리 크기 ~40-100B, 정상 UX 에서 빈도 매우 낮음 (auth-gated) 이나 스크립팅 /
자동화 / paired-device churn 환경에서 N 일 × 수백-수천 엔트리 slow growth.
`critical-devil` 은 추가로 **inFlight promise ref 가 resolve 후에도 잡혀 있을
가능성** 을 drift risk 로 지적했으나 본 솔루션 작성 중 확인: L413-415 finally
가 `state.inFlight = undefined` 로 nullify — 따라서 엔트리당 실측 크기는
`{ lastWakeAtMs: 0 }` number 하나 수준. FIND-002 의 주장 (entry persists +
clearNodeWakeState 가 미등록 nodeId 에 대해 fire 안 함) 은 그대로 유효.

P3: slow leak, OOM-class 아님. 정합성 drift (clearStaleApnsRegistrationIfNeeded
이후 "좀비" 엔트리) 성격 포함.

## 해결 후보

### Candidate 0 — cleanup-on-early-return (chosen)

L334-336 의 no-registration 분기에 `nodeWakeById.delete(nodeId)` 를 추가.

```ts
// BEFORE (nodes.ts:333-336 within the state.inFlight IIFE)
try {
  const registration = await loadApnsRegistration(nodeId);
  if (!registration) {
    return withDuration({ available: false, throttled: false, path: "no-registration" });
  }

// AFTER
try {
  const registration = await loadApnsRegistration(nodeId);
  if (!registration) {
    // Free the entry allocated at L312-313 before returning. The unconditional
    // set was required so concurrent callers share `state.inFlight`; once we
    // decide this nodeId has no APNs registration, the entry has no remaining
    // coalescing value. clearNodeWakeState (L525-528) only fires on WS close
    // for registered-role connections, so unregistered nodeIds would otherwise
    // leak permanently.
    nodeWakeById.delete(nodeId);
    return withDuration({ available: false, throttled: false, path: "no-registration" });
  }
```

**Race 안전성**:
- L312-313 에서 entry 를 set 한 직후 바로 `state.inFlight = (async () => {...})()`
  (L325) 로 채워진 다음 `await state.inFlight` (L412) 로 들어간다. 따라서
  concurrent caller 가 L315 에서 `state.inFlight` 를 보고 같은 promise 를
  `await` 하는 경로는 여전히 가능.
- `no-registration` 분기에서 delete 를 수행하더라도 현재 대기 중인 모든
  caller 의 `await state.inFlight` 는 **promise reference 로 잡혀 있어** 정상
  resolve 됨 (map 에서 지워져도 promise 는 살아 있음). 동일 패턴은 이미
  SOL-0004 의 "in-flight entry evict" corner case 분석에서 확인됨.
- 이후 같은 nodeId 에 새 요청이 들어오면 map miss → 새 state → 새
  loadApnsRegistration. 이는 "미등록 nodeId 에 대한 반복 호출을 throttle 할
  이유 없음" 의미적으로 바른 행동이다 (throttle 은 registration 있는 nodeId 에
  대한 APNs 부하 완화가 목적).

**Tradeoff**: 본체 diff ≈ 3줄 + 주석. 신규 state / field / flag 없음. 기존
flow 의미 변경 최소. `__testing` seam 은 test 지원용 — production behavior
영향 0.

**Scope**: XS.

**Risk**: 매우 낮음. 기존 `nodes.invoke-wake.test.ts` 의 8+ 개 it block 중
no-registration branch 를 테스트하는 것들 (L279, L491, L572, L640) 은 모두
반환 shape (`path: "no-registration"`) 만 assert — map state 는 assert 하지
않으므로 delete 추가로 break 되지 않음.

### Candidate 1 — reorder-check-before-set

`loadApnsRegistration` 결과 확인 **후** `nodeWakeById.set` 수행.

**Tradeoff**:
- FIND-002 root_cause_chain[0] 이 명시한 설계 제약 위배: `state.inFlight` 를
  통한 concurrent wake coalescing 이 unconditional set 에 의존.
- reorder 하면 동일 nodeId 에 대한 N 개 동시 호출이 모두 `loadApnsRegistration`
  을 실행 → 중복 I/O.
- scope M, 의미 변경 영향 넓음 (기존 throttle / dedupe invariant).

→ 기각.

### Candidate 2 — guard-with-hasRegistration

set 전에 APNs registration cache lookup 으로 pre-check.

**Tradeoff**:
- `src/infra/push-apns.ts` 에 cheap has() API 가 없다 (`loadApnsRegistration` 만
  export). 추가 API 도입 → one-thing-per-PR 위배.
- cache lookup 이 sync 가 아니라면 결국 await 중복.

→ 기각.

## 추천

   결과와 무관하게 수행해야 한다는 제약" (state.inFlight coalescing) 을 유지.
