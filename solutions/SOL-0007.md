# cron: mirror activeJobIds mark/clear contract on startup catchup and manual run paths

## 문제 요약

upstream commit `7d1575b5df` (#60310, 2026-04-04) 가 `activeJobIds` 싱글톤 + `markCronJobActive`/`clearCronJobActive` API (`src/cron/active-jobs.ts:15-29`) 를 도입하면서 cron 의 4 개 실행 경로 중 2 개에만 mark/clear 를 주입했다:

| 경로 | 위치 | mark | clear |
|---|---|---|---|
| `runDueJob` | timer.ts:746 | ✓ | ✓ via applyOutcomeToStoredJob (timer.ts:586) |
| `executeJob` | timer.ts:1344 | ✓ | ✓ timer.ts:1374 |
| `runStartupCatchupCandidate` | timer.ts:1043-1081 | **✗ 부재** | n/a (key 없음 → silent no-op) |
| `prepareManualRun` / `finishPreparedManualRun` | ops.ts:548-686 | **✗ 부재** | **✗ 부재** (ops.ts 전체에 active-jobs import 0건) |

task-registry maintenance 의 `hasBackingSession` (`src/tasks/task-registry.maintenance.ts:124-128`) 은 `runtime === "cron"` 일 때 오직 `isCronJobActive(jobId)` 로만 backing evidence 를 판정한다. 두 누락 경로에서 시작된 task 는 ledger 상 `running` 인데 `isCronJobActive=false` → `TASK_RECONCILE_GRACE_MS=5분` (`task-registry.maintenance.ts:28`) 경과 후 `lost`/`missing` 으로 오분류.

### 추가로 grace 가 충분히 통과되는 이유 (cross-review 신규 발견)

- `DEFAULT_JOB_TIMEOUT_MS = 10 * 60_000` (`src/cron/service/timeout-policy.ts:8`) — cron job 은 10 분까지 정상 실행 가능. grace 5 분 < timeout 10 분.
- `recordTaskRunProgressByRunId` 호출이 cron isolated-agent path 에 0 건 — `lastEventAt` 가 startedAt 으로 고정되어 grace reset 안 됨. `hasLostGraceExpired(task, now)` (`task-registry.maintenance.ts:108-111`) 가 reference 로 `lastEventAt ?? startedAt` 사용.

즉 isolated agentTurn (LLM round-trip + 도구 사용) 은 grace 를 정상 초과하므로 재현이 이론적이지 않음.

### Production trigger

- **FIND-001 (startup catchup)**: 매 게이트웨이 부팅마다 `plugins/services.ts:50` (startPluginServices) → `CronService.start` (`cron/service.ts:25-26`) → `ops.start` (`ops.ts:87`) → `runMissedJobs` (`ops.ts:119`) → `executeStartupCatchupPlan` (`timer.ts:971`) → `runStartupCatchupCandidate` (`timer.ts:1043`). missed≥1 시 무조건 진입. `DEFAULT_MAX_MISSED_JOBS_PER_RESTART=5` 로 restart 1회당 최대 5건 노출.
- **FIND-002 (manual run)**: `cron.run` gateway RPC (`server-methods/cron.ts:357`) — CLI (`cron-cli/register.cron-simple.ts:168`), agent tool (`agents/tools/cron-tool.ts:669`), UI controller (`ui/src/ui/controllers/cron.ts:743`) 4 개 진입점이 모두 `context.cron.enqueueRun` → `ops.run` (`ops.ts:688-695`) → `prepareManualRun`/`finishPreparedManualRun` 경로로 수렴.

### 사용자 보고

- **openclaw#68157** (OPEN, 2026-04-23): "Cron isolated agentTurn: already-running survives restart, run history always empty" — 본 SOL 의 정확한 user-visible 증상.
- openclaw#68191 (OPEN, 2026-04-17): contributor `hclsys` 가 같은 root mechanism 발견 + 두 broader 대안 (cron always-backed / durable cron-run store 조회) 제안. 메인테이너 무응답.

### 메인테이너 architecture 가설 (cross-review caveat)

PR #69313 (4/21 merged) 가 `tryRecoverTaskBeforeMarkLost` recovery hook infrastructure 를 신설 (`task-registry.maintenance.ts:315-336`). 단 **cron runtime 은 hook 미등록** — 본 SOL 의 증상은 #69313 만으로 0% 해결됨. 그러나 이 신설 architecture 가 메인테이너의 향후 선호 방향을 시사할 가능성 (CAL-001 패턴 risk: reject 가능). PR description 에 hook 방향 (fix_approach_candidates §C) 을 검토했음을 명시해 redirect 가능성 열어둔다.

async function runStartupCatchupCandidate(
  state: CronServiceState,
  candidate: StartupCatchupCandidate,
): Promise<TimedCronRunOutcome> {
  const startedAt = state.deps.nowMs();
  const taskRunId = tryCreateCronTaskRun({
    state,
    job: candidate.job,
    startedAt,
  });
  emit(state, { jobId: candidate.job.id, action: "started", runAtMs: startedAt });
  // NEW: mirror runDueJob (timer.ts:746) + executeJob (timer.ts:1344) so
  // task-registry maintenance.hasBackingSession sees this run as backed.
  // Pairs with a finally-block clearCronJobActive below — completes the
  // contract introduced by #60310 (7d1575b5df).
  markCronJobActive(candidate.job.id);
  try {
    const result = await executeJobCoreWithTimeout(state, candidate.job);
    return { jobId: candidate.jobId, taskRunId, ...result, startedAt, endedAt: state.deps.nowMs() };
  } catch (err) {
    return { jobId: candidate.jobId, taskRunId, status: "error", error: normalizeCronRunErrorText(err), startedAt, endedAt: state.deps.nowMs() };
  } finally {
    clearCronJobActive(candidate.job.id);
  }
}
