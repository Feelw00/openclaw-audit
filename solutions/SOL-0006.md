      vi.spyOn(

        clientRunId,
        attachments: [sampleImageAttachment()],
        controllerFactory: () => firstController,
      });

        clientRunId,
        attachments: [sampleImageAttachment()],
        controllerFactory: () => secondController,

      // Let both callers pass L1920 and enter parse.
      await microtasksFor(2);

      // Release parse for both.
      parseDeferred.resolve(makeParsedPayload());

      // controller is the one whose run is actually executing. Before the fix,
      // T2's controller overwrites T1's, but T1's run is the one actually

      expect(entry).toBeDefined();
      expect(entry?.controller).toBe(firstController);

      // idempotencyKey during an active run returns {status: "in_flight"}.
      // Before the fix this contract is violated on the attachment branch —
      // the second caller silently proceeds and spawns a duplicate dispatch.
      const ctx = makeSharedChatContext();
      const parseDeferred = createDeferred<ParsedAttachments>();

        "parseMessageWithAttachments",

        clientRunId,
        attachments: [sampleImageAttachment()],
        onRespond: (ok, payload) => responses.push({ t: "t2", ok, payload }),

      parseDeferred.resolve(makeParsedPayload());

      const inFlightAcks = responses.filter(
        (r) => (r as { payload: { status: string } }).payload?.status === "in_flight",
      );
      // Exactly one "started" — the race winner. The other must be "in_flight"
      // per the documented contract.
      expect(startedAcks).toHaveLength(1);
      expect(inFlightAcks).toHaveLength(1);
    });
  });

# gateway/chat.send: post-parse re-check prevents `chatAbortControllers` map overwrite (protect user abort pathway)

## 문제 요약

### 발견

`src/gateway/server-methods/chat.ts:1920-1968` 의 attachment branch 는 sync check → async parsing → sync set 구조. 동일 `clientRunId` 로 두 번째 chat.send 가 parsing window (L1930/L1936 의 `await resolveGatewayModelSupportsImages` + `await parseMessageWithAttachments`) 에 도달하면 **둘 다 L1920 guard 를 통과** → 둘 다 L1960 `chatAbortControllers.set` 을 실행. 최초 주장된 harm: "동일 runId 로 LLM 호출 2회 + session transcript corruption".

```ts
// src/gateway/server-methods/chat.ts:1920 (check)
const activeExisting = context.chatAbortControllers.get(clientRunId);
if (activeExisting) {
  respond(true, { runId: clientRunId, status: "in_flight" }, ...);
  return;
}
// race window 시작 ↓
if (normalizedAttachments.length > 0) {
  const supportsImages = await resolveGatewayModelSupportsImages({ ... }); // L1930 await #1
  const parsed = await parseMessageWithAttachments(..., {                  // L1936 await #2
    maxBytes: 5_000_000, supportsImages,
  });
  ...
}
// race window 끝 ↑
context.chatAbortControllers.set(clientRunId, { controller, ... });        // L1960
respond(true, { runId, status: "started" }, ...);
```

### 방어 로직 확인

Pre-SOL 5-agent confirmation pass 에서 숨은 방어 로직 전수조사 결과 **`claimInboundDedupe`** 가 race 의 agent-pipeline-level 영향을 차단하는 것으로 확인:

- `src/auto-reply/reply/inbound-dedupe.ts:94-112` 가 dedupe key 조합 (`provider|accountId|agent|peerId|threadId|clientRunId`) 에 `clientRunId` 를 포함 (`chat.ts:2028` 의 `MessageSid: clientRunId` 바인딩 경유)
- race 된 두 dispatch 중 두 번째는 `claimInboundDedupe` 에서 `inflight` 판정 → `dispatch-from-config.ts:269-273` 의 `return { queuedFinal: false }` 로 조기 반환
- 결과: **agent pipeline 의 두 번째 invoke 는 차단된다**. LLM 은 1회만 호출된다

이 발견 후 원래 주장 "LLM 호출 2회" / "session transcript corruption" 은 **무효**.

### 일부 방어되나 여전히 문제

`chatAbortControllers.set` (L1960) 은 `claimInboundDedupe` 보다 **상류** (gateway layer) 에 위치. race window 내에서 여전히 발생하는 잔재 피해:

1. **🔴 `chatAbortControllers` map overwrite → user abort 기능 파손** (핵심)
   - T1 run 은 T1 controller 로 실행 중, map 엔트리는 T2 orphan controller 를 가리킴
   - 사용자의 chat 중단 (`chat.abort` RPC) → T2 의 빈 controller 만 abort → 실제 run T1 은 계속 실행
   - **user-visible reliability regression**: 사용자가 중단을 눌러도 작동하지 않는 경로
2. duplicate `{status: "started"}` ack — client 가 동일 runId 에 ack 2개 수신
3. duplicate `persistChatSendImages` / `saveMediaBuffer` fs.writeFile — offloadedRefs 중복 생성 (가벼운 디스크 낭비)
4. `context.dedupe` double-write — T1/T2 의 `.then`/`.catch` 둘 다 같은 `chat:${clientRunId}` 키에 기록

### 공개 API 계약 위반

`docs/web/control-ui.md:131-132`: *"`chat.send` is non-blocking: it acks immediately with `{ runId, status: "started" }` ... Re-sending with the same `idempotencyKey` returns `{ status: "in_flight" }` while running"* — 이 계약은 attachment 가 없을 때만 성립하고 attachment 가 있으면 parsing window 동안 깨진다.

### Production trigger

- macOS `apps/macos/Sources/OpenClaw/GatewayConnection.swift:186-247` 가 URLError 시 `[150, 400, 900]` ms delay 로 동일 params (idempotencyKey 포함) 자동 재전송 (최대 6회)
- Control UI retry storm 사례 — issue #56485
- webchat duplicate observed — issue #46005
- no-attachment branch 는 L1920↔L1960 사이 await 가 0 → JS single-thread atomic 실행 → race 불가 (CAL-001 올바른 guard 예시, 대조군)

## 재현 테스트

frontmatter `repro_test_draft` 참조. 핵심:
- 동일 `context.chatAbortControllers` Map 인스턴스를 T1/T2 공유 (production gateway 는 프로세스당 1 context)
- `parseMessageWithAttachments` 를 deferred Promise 로 mock → race window 확장
- Assertion 1: fix 전에는 T2 controller 가 map 에 남음 / fix 후엔 T1 controller 가 남음 (map 엔트리의 주인이 실제 run 의 주인과 일치)
- Assertion 2: documented `{status: "in_flight"}` 계약이 attachment branch 에서도 지켜짐

이식 위치: **openclaw/src/gateway/server-methods/chat.attachment-race.test.ts (신규)**. 기존 `chat-b.test.ts` / `chat.directive-tags.test.ts` 의 WS harness 는 내부 Map 접근이 안 되어 재활용 불가. `server-node-events.test.ts:50,90,149-170` 의 `buildCtx()` 패턴을 참고해서 `chatAbortControllers` 공유 Map 을 테스트 스코프에서 생성.

CAL-003 compliance: `parseMessageWithAttachments` mock 은 **production 에 실재하는 branch** (attachment.length > 0) 에서 호출되는 함수이며, test 가 production 이 아예 타지 않는 branch 를 강제하는 것이 아님. synthetic-risk 는 "timing 만 인위 확대" 수준이며 race 자체는 production 에 실재 (reproduction-realist 재확인).

## 해결 후보

### A. post-parse re-check (chosen, XS)

```ts
// src/gateway/server-methods/chat.ts 근처 L1958 (parsing 이후, set 직전)
if (normalizedAttachments.length > 0) {
  // ... existing parsing block ...
  // NEW: post-parse re-check — another caller may have won the race during
  // parsing. The documented same-idem retry contract (architecture.md:95-96,
  // control-ui.md:131-132) says an in-flight entry returns {status: in_flight}
  // rather than spawning a duplicate run and overwriting the map entry.
  const duringParseWinner = context.chatAbortControllers.get(clientRunId);
  if (duringParseWinner) {
    respond(true, { runId: clientRunId, status: "in_flight" as const }, undefined, {
      cached: true,
      runId: clientRunId,
    });
    return;
  }
}
// existing L1960 set unchanged
context.chatAbortControllers.set(clientRunId, { ... });
```

- tradeoff: parsing 은 중복 수행됨 (디스크 I/O 약간 낭비, 초과비용 수십 ms). map overwrite / ack 중복 / agent spawn 중복 차단. no-attachment branch 무변경
- diff: ~10줄
- change_scope: XS

### B. parse-inside-work-iife (S)

attachment parsing 을 `chatAbortControllers.set` 이후 (work IIFE 안) 로 이동. no-attachment branch 와 구조가 달라지고, parsing 실패 시 map delete 가 필수 (CAL-001 — cleanup 대칭). diff 중간.

### C. sentinel-pre-insert (M)

L1920 직후 placeholder entry pre-insert, parsing 성공 시 실제 controller 로 교체. `ChatAbortControllerEntry` 타입에 placeholder 의미 추가 필요 + server-maintenance sweep / chat-abort 경로에서 placeholder 처리 추가. 가장 robust 하지만 diff 큼 + 리뷰 부담 큼.

  chatAbortControllers.get guard and the post-parse .set. A same-idempotencyKey

  the initial guard and overwrite the entry, pointing the map at an orphan

  User-initiated chat.abort hits the orphan and fails to stop the real run.

  pipeline invocation (the LLM is only called once). But the gateway-layer
  map overwrite breaks the documented `{status: "in_flight"}` retry contract

  degrades user abort reliability.

  Add a post-parse re-check that returns `{status: "in_flight"}` when a prior
  caller already owns the entry. no-attachment branch is unchanged (atomic
  under JS single-thread semantics; the correct-guard sibling). New regression
  test asserts map entry ownership and contract compliance.

  Related: #69208 (duplicate transcript/replay umbrella — different axis).
  Complementary to: #68801 (agentRunStarts TTL sweep — downstream safety belt).
