# auto-reply queue: drain finally 가 identity 검증 없이 map entry 를 삭제해 후속 queue 를 orphan 시킴

## 문제 요약

`src/auto-reply/reply/queue/drain.ts:263-271` 의 drain IIFE finally 블록은
items=0 && droppedCount=0 일 때 `FOLLOWUP_QUEUES.delete(key)` 와
`clearFollowupDrainCallback(key)` 를 key 만으로 unconditional 실행한다. 자신이
캡처한 `queue` 참조가 현 map entry 와 동일한지 확인하지 않는다.

`/stop` (commands-session-abort.ts:146), session reset
(session-reset-service.ts:258), subagent kill (subagent-control.ts:186),
abort.ts (186/295), gateway sessions.ts:426 등 production caller 다수가
`clearSessionQueues` → `clearFollowupQueue` 를 호출하며, 이는 `queue.items.length = 0`
을 in-place 로 수행 + map.delete 를 unconditional 실행한다 (state.ts:80-86).
그 결과:

- D1 의 `await` (drain.ts:151/161/216/256 등) 중 T1 clearSessionQueues 발생 →
  Q1.items 가 in-place 로 비워지고 map 에서 Q1 제거.
- T2 enqueueFollowupRun(msg2) 가 새 Q2 를 map.set 하고 D2 kick.
- T3 D1 의 await 반환 → loop guard 가 items.length=0 을 보고 자연 종료 →
  finally 의 L266 `FOLLOWUP_QUEUES.delete(key)` 가 **현 map entry 인 Q2 를 제거**.

Impact (wrong-output): `/status` 의 `getFollowupQueueDepth` (status-text.ts:247)
는 map 조회 기반이라 orphan Q2 의 pending 수를 0 으로 보고. 이어서 다른
enqueue 가 들어오면 동일 session key 에 대해 D2/D3 병렬 실행 → per-session
serialization invariant 위반.

production caller (agent-runner.ts:1002) 는 runFollowup 을 항상 전달하므로
callback 손실로 인한 메시지 drop 은 일반 흐름에서 발생하지 않으나, map
invariant 는 깨진 상태.

## 재현 테스트

자세한 코드는 frontmatter `repro_test_draft` 블록. 핵심:

1. `enqueueFollowupRun(msg1)` + `scheduleFollowupDrain` — D1 start, runFollowup
   은 Deferred gate 에서 대기 (production 의 "agent 실행 수 초~분" 시뮬).
2. `clearSessionQueues([key])` — real import. in-place Q1.items.length=0 +
   FOLLOWUP_QUEUES.delete(Q1) + FOLLOWUP_RUN_CALLBACKS.delete.
3. `enqueueFollowupRun(msg2)` — real import. 새 Q2 가 map.set 되고 D2 kick.
4. `gate.resolve()` — D1 의 await 반환 → while 종료 → finally 진입 →
   **unpatched**: L266 이 Q2 를 지움.
5. assert: `FOLLOWUP_QUEUES.get(key) === Q2` (fix 후 통과, 현재 main 실패).

**CAL-003 준수**: mock 없음, real module imports only. Deferred gate 는
production 의 느린 agent 실행을 시뮬하는 realistic device (기존 test 인
`queue.drain-restart.test.ts:207-234` `does not double-drain` 과 동일 패턴).
production 에서 taken 되는 finally L265 branch 를 strange 한 mock 으로
우회 강제하지 않고 실제 `clearSessionQueues` 호출로 자연스럽게 도달.

**이식 목적지**: `src/auto-reply/reply/queue/drain.identity-guard.test.ts`
(default vitest 프로필). 신규 파일로 분리 — regression 기대 단언이 map
identity 에 focus 되며 기존 `queue.drain-restart.test.ts` 의 scope (callback
restart 경로) 와 다름. 메인테이너가 같은 파일에 넣자고 요청하면 즉시 반영
가능.

## 해결 후보

### Candidate 0 — add-identity-guard (chosen)

drain.ts:263-271 finally 블록을 identity guard 로 감싸 map entry 가 여전히
자신의 `queue` 인스턴스일 때에만 delete + clearCallback 수행.

```ts
// BEFORE (drain.ts:263-271)
} finally {
  queue.draining = false;
  if (queue.items.length === 0 && queue.droppedCount === 0) {
    FOLLOWUP_QUEUES.delete(key);
    clearFollowupDrainCallback(key);
  } else {
    scheduleFollowupDrain(key, effectiveRunFollowup);
  }
}

// AFTER
} finally {
  queue.draining = false;
  if (queue.items.length === 0 && queue.droppedCount === 0) {
    // Identity guard: only dispose map entry/callback if no external clear
    // (clearFollowupQueue) or subsequent enqueue replaced the entry. Without
    // this check a late finally could orphan a newly registered queue.
    if (FOLLOWUP_QUEUES.get(key) === queue) {
      FOLLOWUP_QUEUES.delete(key);
      clearFollowupDrainCallback(key);
    }
  } else {
    scheduleFollowupDrain(key, effectiveRunFollowup);
  }
}
```

**효과**:
- 정상 drain (외부 clear 없음) → `FOLLOWUP_QUEUES.get(key) === queue` true →
  기존과 동일하게 delete (기존 테스트 `clears the remembered callback after a
  queue drains fully` 그대로 통과).
- 외부 clear + Q2 등록 후 late finally → `FOLLOWUP_QUEUES.get(key) === Q2 !== Q1`
  → skip. Q2 map entry / Q2 callback 보존.
- 외부 clear 후 Q2 등록 **없음** → `FOLLOWUP_QUEUES.get(key)` 는 undefined →
  `undefined === Q1` false → skip (noop, 이미 map 에 없음).

**Tradeoff**: diff 2줄. 신규 state/필드/플래그 없음. 기존 의미 변경 최소.
**Scope**: XS.
**Risk**: 매우 낮음.

### Candidate 1 — move-delete-to-kickFollowupDrainIfIdle

finally 의 delete 를 제거하고 map.delete 를 외부 clear + enqueue 의 idle
감지로 이관. 개념상 더 깔끔하나:
- 기존 테스트 다수가 "drain 완료 후 map 에서 자동 제거" 를 검증
  (queue.drain-restart.test.ts:14-38, 262-283).
- observer 의미 변경 — 예: drain 완료 직후 `getFollowupQueueDepth` 가 0 대신
  빈 queue 존재로 보일 가능성.
- scope M, regression risk medium.

→ 이번 PR 스코프 아님.

### Candidate 2 — version-counter

`FollowupQueueState.generation: number` 추가 + recreate 시 증분 + finally
에서 generation 비교. identity guard 와 기능적으로 동등하지만 추가 필드
+ 증분 로직 + re-enqueue 핸들링. CAL-001 의 defense-in-depth 회피 교훈에
어긋남 (간단한 `===` 비교로 충분한 곳에 새 메커니즘).

→ over-engineering. 거부.

## 추천

**Candidate 0 (chosen_fix=0)** — add-identity-guard.

근거:
1. **XS scope** — 2줄 조건부 guard 추가. 메인테이너 가이드의 "XS 선호" 부합.
2. **기존 패턴 유지** — 동일 패턴이 `src/agents/subagent-announce-queue.ts:62-64`
   주석에 maintainer 가 인지하고 있음 ("Clearing the map alone isn't enough
   because drain loops capture queue by reference"). 본 fix 는 production
   finally 경로에 그 사유를 그대로 적용.
3. **CAL-001 defense-in-depth 회피** — 새 state / flag / counter 도입 없이
   map 의 자기 일관성만 사용. CAND-011/SOL-0002 가 필요했던 `browserCleanupDispatchedAt`
   같은 새 필드가 여기서는 불필요 (maps 은 이미 identity 로 비교 가능한 객체).
4. **CAL-003 반영** — 재현 테스트가 production hot-path (clearSessionQueues
   real 호출 → finally L265 true branch) 와 동일 branch 를 exercise. mock 으로
   강제된 경로 없음.
5. **Regression 위험 0** — 정상 drain 의 identity 는 자기 자신과 일치하므로
   기존 delete 의미 그대로 유지. 기존 테스트 queue.drain-restart.test.ts 의
   8개 it block 모두 Q1 의 lifecycle 내에서 진행 → identity always matches.

**커밋 메시지 (초안)**:

```
fix(queue): guard drain finally map delete with queue identity check

scheduleFollowupDrain 의 drain IIFE 는 finally 에서 items=0 && dropped=0 일 때
FOLLOWUP_QUEUES.delete(key) 와 clearFollowupDrainCallback(key) 를 key 만으로
호출한다. 자신이 캡처한 queue 참조가 현재 map entry 와 동일한지 검증하지
않으므로, drain 의 await 중 /stop, session reset, subagent kill 등이
clearSessionQueues -> clearFollowupQueue 를 호출하고 이후 enqueueFollowupRun
이 새 queue 를 등록하면, 원래 drain 의 late finally 가 새 queue 의 map
entry 와 callback 을 지워 orphan 시킨다. getFollowupQueueDepth 가 실제
pending 을 0 으로 보고하고 (status-text.ts:247), 동일 session key 에 대해
두 drain 이 병렬 실행될 수 있다.

finally 에서 FOLLOWUP_QUEUES.get(key) === queue 일 때에만 delete + clearCallback
을 수행하도록 identity guard 추가. 정상 drain 의 의미는 변경되지 않는다.

동일 "capture queue by reference" 패턴은 subagent-announce-queue.ts:62-64
주석에 이미 기록되어 있으며 이번 fix 는 production finally 경로에 그 사유를
적용한다.

Closes #NNN
```
