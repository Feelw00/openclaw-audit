# gateway/usage: costUsageCache 에 MAX + FIFO eviction 도입 (sibling pattern 복제)

## 문제 요약

`src/gateway/server-methods/usage.ts:65` 의 모듈-레벨 `costUsageCache: Map<string, CostUsageCacheEntry>` 는 `(startMs, endMs)` 쌍을 key 로 `CostUsageSummary` 결과를 보관한다. TTL(`COST_USAGE_CACHE_TTL_MS = 30_000`, L51) 은 L310 read-path 에서 stale 값을 "읽지 않을" 뿐이며, 엔트리를 Map 에서 **제거하는 production 경로는 존재하지 않는다** (Grep `costUsageCache.(delete|clear|evict|splice|shift|pop)` → 프로덕션 match 0). `parseDateRange` 는 `getTodayStartMs(now, interpretation)` (L191-203) 기반이므로 UTC 00:00 마다 cacheKey 가 하루씩 전진하고, `days`/`startDate`/`endDate`/`utcOffset` 가변 파라미터가 cardinality 를 곱한다.

macOS menu (`MenuSessionsInjector.swift:36,75,128`) 는 `usage.cost` 를 params=nil 로 **45초 폴링** 하고 control-channel reconnect 마다 추가 호출 → `parseDateRange({})` default branch → 매일 새 cacheKey 생성 → 이전 key 는 영구 잔류. Map 은 `O(uptime_days × range_options × utc_variants)` 로 monotonic 증가한다.

동일 파일 권역의 sibling cache 3 건은 모두 MAX + FIFO eviction 을 구현:

- `resolvedSessionKeyByRunId` — `src/gateway/server-session-key.ts:16, 22-34`, `RUN_LOOKUP_CACHE_LIMIT=256`
- `TRANSCRIPT_SESSION_KEY_CACHE` — `src/gateway/session-transcript-key.ts:141-150`, `MAX=256`, 주석 "Evict oldest-inserted entry when cache exceeds size cap (FIFO bound)"
- `sessionTitleFieldsCache` — `src/gateway/session-utils.fs.ts:28, 57-69`, `MAX=5000`, while-loop FIFO

`costUsageCache` 만 이 패턴이 누락이다.

## 재현 테스트

자세한 코드는 frontmatter `repro_test_draft`. 핵심 흐름:

1. `__test.costUsageCache.clear()` 로 시작 (usage.test.ts:27 기존 패턴 그대로).
2. `loadCostUsageSummary` 는 top-level `vi.mock` 으로 경량 stub (usage.test.ts:4-18 동일 패턴).
3. `for (i=0; i<600; i++)` 로 `(startMs, endMs)` 쌍을 매 iteration 마다 `i * DAY_MS` 만큼 전진 + `days` 스위치 효과를 `i % 3` 으로 재현 → 600개 distinct key 생성. 이 규모는 sibling 캐시 중 가장 작은 cap (256) 보다 크고 가장 큰 cap (5000) 보다 작음.
4. assertions:
   - `costUsageCache.size < ITERATIONS` — 현 main 에선 `=== 600` 이라 실패. fix 후엔 cap 로 plateau.
   - 마지막 insert key 는 여전히 존재 (FIFO 는 가장 오래된 것을 evict).
   - 첫 insert key 는 evict 되어 부재 (cap 초과 시).

**CAL-003 준수**: 내부 branch 를 강제하는 mock 없음. `loadCostUsageSummary` stub 은 이미 usage.test.ts 상단에서 쓰이는 동일 경량 stub (external IO 경계 mock 만, 내부 branch 조작 0). key 증가는 production 과 동일한 input 가변성 (날짜/days) 으로 자연 발생.

**이식 목적지**: `src/gateway/server-methods/usage.cost-usage-cache.test.ts` (default vitest 프로필). 신규 파일 분리 근거는 관심 구분 — 기존 `usage.test.ts` 는 `parseDateRange` / `parseDateToMs` / TTL hit 경로 검증에 집중. bounded-growth regression 은 별개 단언 축. 메인테이너가 같은 파일 선호 시 즉시 병합 가능.

## 해결 후보

### Candidate 0 — mirror-sibling-fifo (chosen)

set 경로 3곳 (L328, L341, L346) 중 "새 cacheKey 가 처음 삽입되는 L346" 에 cap+FIFO guard 추가. L328 과 L341 은 **동일 key 갱신**이므로 새 엔트리를 만들지 않아 guard 불필요.

```ts
// BEFORE (usage.ts 전역)
const COST_USAGE_CACHE_TTL_MS = 30_000;
const costUsageCache = new Map<string, CostUsageCacheEntry>();

// AFTER
const COST_USAGE_CACHE_TTL_MS = 30_000;
const MAX_COST_USAGE_CACHE_ENTRIES = 256;
const costUsageCache = new Map<string, CostUsageCacheEntry>();

// usage.ts: loadCostUsageSummaryCached, L345-346 근처
//   entry.inFlight = inFlight;
//   costUsageCache.set(cacheKey, entry);
// 를 아래처럼 교체:

entry.inFlight = inFlight;
// Evict oldest-inserted entry when cache exceeds size cap (FIFO bound).
// Mirrors resolvedSessionKeyByRunId (server-session-key.ts:22-30),
// TRANSCRIPT_SESSION_KEY_CACHE (session-transcript-key.ts:141-150),
// and sessionTitleFieldsCache (session-utils.fs.ts:63-69).
if (
  !costUsageCache.has(cacheKey) &&
  costUsageCache.size >= MAX_COST_USAGE_CACHE_ENTRIES
) {
  const oldest = costUsageCache.keys().next().value;
  if (oldest !== undefined) {
    costUsageCache.delete(oldest);
  }
}
costUsageCache.set(cacheKey, entry);
```

**MAX 값 선정 근거**: `TRANSCRIPT_SESSION_KEY_CACHE_MAX = 256` 및 `RUN_LOOKUP_CACHE_LIMIT = 256` 와 정합. `sessionTitleFieldsCache` 의 5000 은 per-file 기반이라 cardinality 가 다름 — cost cache 는 date-range cardinality (일·days·utcOffset 조합) 로 256 엔트리면 1년 daily rollover × 30-option range × 변주 충분 커버. PR #36682 이 제안한 64 보다 넉넉해 리뷰 반론 여지 감소.

**왜 L346 한 곳이면 충분한가**: L328 (inFlight resolve 시 summary set) / L341 (finally 의 inFlight 해제) 는 기존 cacheKey 를 `has()` → true 로 갱신 — Map.size 증가 없음. 새 entry 삽입은 오직 L346 (miss + 새 inFlight 시작 경로) 에서만 발생. 따라서 eviction guard 는 L346 앞에만 두면 size 상한이 성립.

**Tradeoff**: diff ≈ 10줄 (상수 1 + guard block 1). 신규 state / 필드 / flag 없음. 기존 flow 의미 100% 보존 (read-path TTL, in-flight dedup, same-key overwrite 모두 동작 그대로).

**Scope**: XS.

**Risk**: 매우 낮음. `usage.test.ts:141-160` TTL 테스트는 cacheKey 1개만 사용 → cap 과 무관. 새 regression test 는 별 파일.

### Candidate 1 — ttl-on-write

set 시점에 `costUsageCache.entries()` 순회 → `updatedAt + N * COST_USAGE_CACHE_TTL_MS` 초과 엔트리 delete.

**Tradeoff**:
- sibling 3건이 전부 MAX+FIFO 라서 왜 이 cache 만 다른 방식인지 설명 부담. 메인테이너 리뷰에서 "make it consistent" 요청 가능성 높음 (한 PR 에 non-trivial scope 추가 → 재작업).
- TTL 기반 prune window (2× TTL = 60초? 10× TTL = 5분?) 에 arbitrary choice — 버킷 선정 근거 취약.
- in-flight 중인 entry (summary 없음, inFlight 있음) 가 TTL 기준에 섞이지 않게 guard 필요 → 분기 증가.
- 테스트 시나리오도 까다롭다: fake timer 로 TTL 경계 조작 → usage.test.ts 의 `useRealTimers()` 수렴 패턴과 충돌 가능.

→ 기각. sibling-consistency 점수에서 밀린다.

### 2. PR #36682 가 LRU 를 선택한 이유 재검토

promote-on-hit (LRU) 는 "자주 접근하는 key 를 오래 보존" 이점. 그러나 costUsageCache 의 access pattern 은:
- Same-key 재요청은 대부분 **TTL 30초 내 동일 macOS polling** → TTL hit 이지 Map promotion 불필요.
- TTL 밖 재요청은 stale 갱신 — FIFO 든 LRU 든 cache miss 비용 동일.
- "어제의 key 는 절대 다시 조회되지 않음" (todayStartMs 가 바뀌었으므로) — LRU 의 promotion 이득 거의 없음.

→ FIFO 로 충분. LRU 의 `delete-then-set` 으로 인한 매 hit per-op 비용 (amortized O(1) 이나 cache-line churn) 회피. sibling cache 들이 FIFO 인 것도 같은 이유로 추정.

### 3. in-flight entry 가 evict 될 가능성

L346 에서 `!costUsageCache.has(cacheKey) && size >= MAX` 조건에서 oldest 를 delete 할 때, oldest 가 "summary 없이 inFlight 만 있는 race entry" 일 가능성. 그러나:
- in-flight promise 는 `cached.inFlight` (L314) 참조로 closure 에 붙어 있어 Map 에서 삭제돼도 promise 자체는 계속 resolve 됨 (GC 는 closure chain 으로 살아 있음).
- L338-341 의 `.finally` 에서 `costUsageCache.get(cacheKey)` 가 undefined 여도 `if (current?.inFlight === inFlight)` 가 false 가 되어 early return — 에러 없음.
- 단점: 나중에 해당 key 가 **재요청** 되면 L309 `cached = undefined` → cache miss → 새 inFlight 시작. 이는 동일 요청이 잠깐 중복 실행될 수 있다는 의미 (drop 은 아님). 극한 상황 (MAX 직전까지 in-flight 만 있는 경우) 만 해당 → 실무상 무시 가능.

→ 이 corner case 는 test 에 포함시키지 않음 (과잉 세밀도). PR 본문 "behavior" 섹션에 한 줄 언급.
