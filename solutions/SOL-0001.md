# sweeper self-stop: pending lifecycle error TTL cleanup 무력화 방지

## 문제 요약

`src/agents/subagent-registry.ts` 의 sweeper (60초 주기) 가 매 사이클 종료 시 `subagentRuns.size === 0` 이면 자기 자신을 `stopSweeper()` 호출. 그러나 `pendingLifecycleErrorByRunId` Map 은 별도 TTL(5분) 기반 cleanup 을 같은 sweeper 에서 수행 (line 595-599). subagentRuns 가 비어 있지만 pending error entry 가 남아 있는 순간에 sweeper 가 stop 되면, pending map 의 TTL cleanup 자체가 발동 안 함 → 다음 `registerSubagentRun` (idle 워크로드에서는 수 시간~수 일 후) 까지 entry + NodeJS.Timeout 영구 상주.

**origin/main 기준 코드 위치**:
- sweeper body self-stop: `src/agents/subagent-registry.ts:604-606`
- pending TTL 루프: `:595-599`
- stopSweeper 정의: `:524-530`
- startSweeper 정의: `:511-522`
- schedulePendingLifecycleError: `:242`
- PENDING_ERROR_TTL_MS=5m: `:140`
- pending map 선언: `:217`

## 재현 테스트

자세한 코드는 frontmatter `repro_test_draft` 블록 참조. 핵심 시나리오:

1. `registerSubagentRun(runId=r1)` → sweeper 가동 (line 717 경유 startSweeper)
2. lifecycle error 이벤트 → `schedulePendingLifecycleError(r1)` → pending map 에 entry 추가
3. 다른 경로로 `subagentRuns.delete(r1)` (orphan recovery 등) → subagentRuns 비어짐, pending 남음
4. sweeper 한 사이클 진행 (60초 advance) → pending TTL 미도달(5분 미경과) 이라 cleanup 안 함 → self-stop (line 604)
5. 추가 5분 advance → 정상 동작이면 sweeper 가 다시 돌아 cleanup 해야 함. 현재는 sweeper 가 stop 이라 미청소
6. pending map 크기 > 0 확인 → **현재 failing** (정상 동작이면 0)

수정 후: sweeper 가 pending map 에 entry 있는 한 멈추지 않으므로 TTL 만료 시점의 다음 cycle 에서 cleanup.

**이식 목적지**: `src/agents/subagent-registry.sweeper-self-stop.test.ts` (default vitest 프로필).
**테스트 인프라 확장 필요**: `src/agents/subagent-registry-test-helpers.ts` 에 3개 accessor 추가:
- `getSubagentRunsSizeForTest()`
- `getPendingLifecycleErrorByRunIdSizeForTest()`
- `triggerLifecycleErrorEventForTest(runId)`

PR 범위에 helpers 변경 포함. test-only export 이므로 public API 영향 없음.

## 해결 후보

### Candidate 0 — extend-self-stop-condition (chosen)

```ts
// 현재 (line 604-606):
if (subagentRuns.size === 0) {
  stopSweeper();
}

// 수정:
if (subagentRuns.size === 0 && pendingLifecycleErrorByRunId.size === 0) {
  stopSweeper();
}
```

- **Tradeoff**: sweeper 가 pending drain 될 때까지 최대 ~5~6분 추가 회전. setInterval `.unref()` 돼있으니 process exit 막지 않음. 의미론이 자연스러움 ("청소할 일 남아있으면 청소부도 남는다").
- **Scope**: 1줄 조건 확장. XS PR.
- **Risk**: 매우 낮음. `pendingLifecycleErrorByRunId` 는 이미 같은 모듈 스코프에서 접근 중.

### Candidate 1 — drain-on-stop

`stopSweeper()` 에 `clearAllPendingLifecycleErrors()` 추가. 즉시 drain. 단 15초 grace timer 가 들고 있는 pending.error 내용이 즉시 사라져 `completeSubagentRun` 의 비정상 종료 통보가 drop 될 가능성. 1번보다 위험.

### Candidate 2 — ensure-sweeper-in-schedule

`schedulePendingLifecycleError` 에서 sweeper stopped 면 재가동. 1번과 상호 보완. 단 sweeper lifecycle 이 3곳(registerSubagentRun + schedulePendingLifecycleError + runSubagentRunSweep restore) 분산 → 불변식 추적 어려움.

## 추천

**Candidate 0 (chosen_fix=0)** 단독 채택. XS, 의미론 직관적, 위험 낮음.

Candidate 2 는 follow-up 으로 별도 PR 가치 있음 (pending-only 경로 재가동 — Candidate 0 이 커버 못하는 "sweeper 가 이미 stopped 된 상태에서 새 pending 이 들어오는" 경쟁 조건). 단 본 PR 에는 포함 안 함 — one thing per PR.
