# subagent-registry: guard cleanupBrowserSessionsForLifecycleEnd with beginSubagentCleanup

## 문제 요약

`src/agents/subagent-registry-lifecycle.ts:635-644` 의 `completeSubagentRun` 은 `cleanupBrowserSessionsForLifecycleEnd` (line 639) 를 `startSubagentAnnounceCleanupFlow` (line 644) 이전에 호출한다. 원자 가드 `beginSubagentCleanup` (lifecycle.ts:280-291) 은 `startSubagentAnnounceCleanupFlow` 내부 (lifecycle.ts:476, 492) 에서만 taken 되므로, line 639 시점에는 `entry.cleanupHandled` 가 false.

`registerSubagentRun` (run-manager.ts:348, 354) 이 embedded mode 에서 `ensureListener()` 와 `void waitForSubagentCompletion(runId, ...)` 을 **항상 둘 다** 기동하고, 두 경로 모두 `triggerCleanup:true` 로 `completeSubagentRun` 을 호출하므로 동일 runId + 동일 `childSessionKey` 에 대한 브라우저 세션 정리 래퍼가 2회 invoke 된다.

**중복 스코프 (pre-pr cross-review 후 정정)**:

| 계층 | 2회 실행되는가? |
|---|---|
| `cleanupBrowserSessionsForLifecycleEnd` wrapper call | ✅ 예 |
| `normalizeSessionKeys` + `runBestEffortCleanup` | ✅ 예 |
| `plugin-sdk` facade resolution (`closeTrackedBrowserTabsForSessions` 동적 load) | ✅ 예 (cold path 비용 있음) |
| `onWarn` 로그 경로 (empty tabs 경우) | ✅ 예 |
| **실제 CDP `browserCloseTab` IPC** | ❌ **아니오** — `session-tab-registry.ts:129` 의 `takeTrackedTabsForSessionKeys` 가 `trackedTabsBySession.delete(sessionKey)` 로 drain → 두 번째 호출은 line 148 `tabs.length === 0` early-return |

→ 이 fix 는 wrapper-level 중복 dispatch 를 차단하는 것이며, 실제 CDP IPC 는 이미 bundled browser extension 의 take-and-drain 패턴으로 dedup 되어 있다. 따라서 **defense-in-depth + wrapper overhead 절감** 성격이 강하다. 그럼에도 다음 이유로 가치가 있다:
- 48042c3875 `endedHookEmittedAt` 선행 fix 와 동일 파일에서 동일 dedup 패턴 확장 — 설계 일관성
- plugin-sdk facade lazy load 비용 제거
- 향후 bundled browser 구현이 take-and-drain 을 제거하거나 교체하면 이 layer 가 primary guard 가 됨

## 해결 후보

### Candidate 0 — dedicated-dispatch-flag (chosen)

`SubagentRunRecord` 에 `browserCleanupDispatchedAt?: number` 필드를 추가하고, `completeSubagentRun` 의 browser cleanup 호출 직전에 check-then-set 가드를 배치. 48042c3875 의 `endedHookEmittedAt` 과 동일 패턴.

**주의**: 두 번째 호출자가 `return;` 으로 early-exit 하므로 `startSubagentAnnounceCleanupFlow` 도 실행되지 않는다. 이게 맞는가?
- `startSubagentAnnounceCleanupFlow` 는 내부적으로 `beginSubagentCleanup` 로 이미 dedup 됨 (cleanupHandled)
- 두 호출자 중 하나만 announce flow 를 시작하고, 나머지는 skip 해야 정상 — 그래서 `return;` 은 의도대로
- 실질적으로 browser cleanup + announce cleanup 을 한 호출자가 함께 수행하고, 나머지는 skip

### Candidate 1 — move-cleanup-inside-flow

Browser cleanup 을 `startSubagentAnnounceCleanupFlow` 내부로 이동. 단 `startSubagentAnnounceCleanupFlow` 가 `retryDeferredCompletedAnnounces` (lifecycle.ts:310) 에서도 호출되므로, browser cleanup 이 retry 시점에도 새로 호출됨 → 의미 변경.

→ intent 확인 필요. 메인테이너 질문 유발 가능성. 이번 PR 스코프 아님.

### Candidate 2 — guard-at-completeSubagentRun-entry-with-cleanupHandled

`beginSubagentCleanup(runId)` 을 line 638 에 직접 호출. 첫 통과자가 `cleanupHandled=true` 세팅 → `startSubagentAnnounceCleanupFlow` 내부 `beginSubagentCleanup` false → announce cleanup skip (regression).

→ 부적합 (announce cleanup 손상).

### Candidate 3 — dedupe-at-caller

`registerSubagentRun` level 에서 inflight set 도입. 근본적이지만 scope M, 기존 `endedHookEmittedAt`/`inFlightRunIds` 와 책임 중복.

→ follow-up 으로 분리 가능.
