# fix(mcp): bound pendingClaudePermissions / pendingApprovals via TTL sweeper + close clear + cap

## 문제 요약

`OpenClawChannelBridge` (`src/mcp/channel-bridge.ts`, 516 라인) 의 두 instance-bound Map — `pendingClaudePermissions` (L50) 과 `pendingApprovals` (L51) — 은 같은 클래스의 다른 컬렉션 (`queue` L48 의 `QUEUE_LIMIT=1000` cap, `pendingWaiters` L49 의 close-clear + per-waiter `setTimeout`) 과 달리 **TTL · cap · close-clear 세 가드를 동시에 결여**한다. set 은 외부 trigger (Claude SDK `notifications/claude/channel/permission_request`, gateway `exec/plugin.approval.requested` EventFrame) 마다 unconditional 으로 발생 (`L279`, `L374`) 하는 반면 delete 는 conditional-edge 단일 경로 (정규식 매칭+sessionKey 검증, gateway resolved 이벤트 도달) 에만 의존한다. 미응답 / 형식 오탈 / sessionKey mismatch / WS drop / gateway server-side timeout 시 entry 가 영구 잔존하며, `close()` (L136-152) 가 두 Map 을 비우지 않고 클래스 어디에도 `setInterval` sweeper 가 없어 시간 기반 정리도 0 건. `expiresAtMs` (L382) 는 entry 에 저장만 되는 read-only 메타다. `openclaw mcp serve` 가 hours-units long-running stdio 프로세스로 가동되는 의도와 결합되면 외부 trigger 빈도에 비례한 단조 증가 leak 이 성립한다.

upstream HEAD `c070509b7f` 직접 검증: 6주 channel-bridge.ts 커밋 (`e157c83c65`/`0f7d9c9570`/`74e7b8d47b`/`ba02905c4f`/`ec5877346c`/`71f37a59ca`) 모두 refactor / seam 분리 / 스모크 강화로 leak 가드 추가 0건. PR #56420 (OPEN, sessionKey binding) 은 보안 spoofing 축으로 직교 — 본 fix 와 의미 충돌 없으나 close() 본문에서 라인 충돌 가능 (rebase merge 수준).

### A. sweeper+ttl-only (XS, sweeper interval + per-Map TTL, close-clear + clearInterval 포함)

```ts
// src/mcp/channel-bridge.ts
const QUEUE_LIMIT = 1_000;
const CLAUDE_PERMISSION_TTL_MS = 10 * 60_000;        // NEW
const PENDING_SWEEP_INTERVAL_MS = 30_000;            // NEW

interface PendingClaudePermissionEntry extends ClaudePermissionRequest {
  createdAtMs: number;                                // NEW
}

export class OpenClawChannelBridge {
  // ... 기존 ...
  private readonly pendingClaudePermissions = new Map<string, PendingClaudePermissionEntry>();
  private sweeperInterval: ReturnType<typeof setInterval> | null = null;  // NEW

  constructor(/* ... */) {
    // ... 기존 ...
    // NEW: long-running stdio process 의도 (hours-units) 와 정합 — 두 Map
    // 의 stale entry 정리. unref() 로 process keep-alive 부수효과 차단.
    this.sweeperInterval = setInterval(
      () => this.sweepPendingMaps(this.cfg.now?.() ?? Date.now()),
      PENDING_SWEEP_INTERVAL_MS,
    );
    this.sweeperInterval.unref?.();
  }

  private sweepPendingMaps(nowMs: number): void {
    // FIND-001: createdAtMs 기준 TTL 초과 시 evict.
    for (const [requestId, entry] of this.pendingClaudePermissions) {
      if (nowMs - entry.createdAtMs > CLAUDE_PERMISSION_TTL_MS) {
        this.pendingClaudePermissions.delete(requestId);
      }
    }
    // FIND-002: payload.expiresAtMs 가 entry 에 이미 있음 (L382) — 그 값 활용.
    for (const [id, entry] of this.pendingApprovals) {
      if (typeof entry.expiresAtMs === "number" && entry.expiresAtMs <= nowMs) {
        this.pendingApprovals.delete(id);
      }
    }
  }

  async handleClaudePermissionRequest(params: { /* ... */ }): Promise<void> {
    this.pendingClaudePermissions.set(params.requestId, {
      toolName: params.toolName,
      description: params.description,
      inputPreview: params.inputPreview,
      createdAtMs: Date.now(),                        // NEW
    });
    // ... 기존 enqueue ...
  }

  async close(): Promise<void> {
    if (this.closed) return;
    this.closed = true;
    this.resolveReadyOnce();
    if (this.sweeperInterval) {                       // NEW
      clearInterval(this.sweeperInterval);
      this.sweeperInterval = null;
    }
    for (const waiter of this.pendingWaiters) {
      if (waiter.timeout) clearTimeout(waiter.timeout);
      waiter.resolve(null);
    }
    this.pendingWaiters.clear();
    this.pendingClaudePermissions.clear();            // NEW (FIND-001)
    this.pendingApprovals.clear();                    // NEW (FIND-002)
    const gateway = this.gateway;
    this.gateway = null;
    await gateway?.stopAndWait().catch(() => undefined);
  }
}
```

- **Diff**: ~30-50 라인 (constants + sweeper method + constructor + close 변경 + ClaudePermissionRequest entry 확장).
- **이유**: hot-path-tracer 권고 우선순위 (sweeper+TTL > close-clear > cap) 정렬. close-clear 만 단독으로는 process exit only 회수라 실효 작음 — sweeper 가 진짜 fix. `sweeperInterval.unref()` 로 process keep-alive 부수효과 방지. `clearInterval` 을 close() 의 일부로 명시 — timer leak 자체로 또 다른 leak 만들지 않음.
- **약점**: cap 부재 — 적대적 polling burst (Claude SDK 가 30s 안에 1000+ permission_request 보내는 비정상 시나리오) 시 sweeper tick 사이 누적. 정상 운영에서 실효 위험 작음.
